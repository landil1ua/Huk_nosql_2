## Частина 1. Вибір інструментів

### Pinecone, Qdrant, Chroma

**Pinecone** я використав, бо це частина завдання і з ним не треба піднімати локальний сервер. Достатньо створити індекс у хмарі і працювати через API.

**Qdrant** теж міг би підійти. Його можна запускати локально або на своєму сервері, тому він був би зручний, якщо треба більше контролю над даними.

**Chroma** більше схожа на варіант для локальних експериментів. Для маленьких тестів це зручно, але для цього завдання Pinecone виглядає ближче до продакшн-варіанту.

Якщо коротко:

| Інструмент | Коли зручно |
|---|---|
| Pinecone | коли треба швидко зробити хмарний векторний пошук |
| Qdrant | коли треба self-hosted або більше контролю |
| Chroma | коли треба простий локальний прототип |

### Чому `allenai/specter2_base`

Я вибрав `allenai/specter2_base`, бо це модель саме для наукових статей. У нас якраз дані з arXiv: заголовок + анотація. Для такої задачі спеціалізована модель логічніша, ніж загальна модель типу `all-MiniLM-L6-v2`.

Текст для ембеддингу я формував так:

```text
title [SEP] abstract
```

Це підходить для SPECTER-подібних моделей, бо вони якраз орієнтовані на роботу з title/abstract.

`all-MiniLM-L6-v2` теж можна було б взяти, але вона більш універсальна. Вона добре працює для звичайних речень, але для наукових текстів я очікував кращий результат від `specter2_base`.

### Метрика

У Pinecone я використав `cosine`. Ембеддинги нормалізуються в `02_embed.py` через:

```python
normalize_embeddings=True
```

Для нормалізованих векторів cosine і dot product дають однакове ранжування, бо довжина векторів дорівнює 1.

Формула cosine:

```text
cos(a, b) = (a · b) / (||a|| · ||b||)
```

Якщо `||a|| = 1` і `||b|| = 1`, то залишається просто:

```text
cos(a, b) = a · b
```

Тому на нормалізованих векторах cosine і dot product майже еквівалентні для пошуку.

### Пошукові запити

У `04_search.py` я зробив:

1. звичайний семантичний пошук;
2. пошук з фільтром за роком і категорією;
3. локальне порівняння метрик.

Основний запит:

```text
teaching machines to recognize objects in pictures
```

Для нього в топі були статті про image recognition / computer vision, наприклад `Butterfly Detection and Classification Based on Integrated YOLO Algorithm`.

Також були два фільтровані приклади:

```text
reinforcement learning
```

- статті після 2021 року в категорії `cs.LG`;
- статті до 2015 року.

Фільтри тут корисні, бо один і той самий запит можна обмежити потрібним роком або категорією. Наприклад, для `reinforcement learning` можна окремо дивитись нові роботи або старіші статті.

### Порівняння cosine, dot product і L2

Я локально порівняв три метрики:

- cosine similarity;
- dot product;
- L2 distance.

На нормалізованих векторах топ-5 для всіх трьох метрик вийшов однаковий.

Причина:

```text
||a - b||² = 2 - 2*cos(a, b)
```

Якщо вектори нормалізовані, то чим більший cosine, тим менша L2-відстань. Тому порядок документів виходить той самий.

Якби вектори не нормалізувати, результати могли б відрізнятися. Dot product тоді залежав би не тільки від напрямку, а ще й від довжини вектора. L2 теж почала б враховувати різницю в довжинах.

## Частина 4. Chunking

У `05_chunking.py` я взяв 30 статей з найдовшими анотаціями і розбив їх двома способами.

### Fixed-size chunking

Параметри:

```python
CHUNK_SIZE = 100
OVERLAP = 20
```

Тобто кожен чанк має приблизно 100 слів, а сусідні чанки перетинаються на 20 слів.

Плюс цього способу — він дуже простий. Мінус — він може розривати речення посередині.

### Semantic chunking

Тут я ділю текст по реченнях і збираю чанки до приблизно 150 слів.

Плюс — чанки виходять більш цілісні, бо речення не ріжуться. Мінус — розмір чанків виходить менш рівномірний.

### Індекси

Для chunking створені два окремі індекси:

```text
arxiv-chunks-fixed
arxiv-chunks-semantic
```

У логах вийшло:

```text
Fixed чанків: 122
Semantic чанків: 71
```

Semantic-чанків менше, бо вони часто довші.

### Висновок по chunking

Мені більше сподобався semantic chunking, бо текст у результатах читається природніше. Fixed-size chunking теж працює, але іноді шматок починається або закінчується не дуже вдало.

Overlap у fixed-size потрібен для того, щоб інформація на межі двох чанків не губилася. Але чим більший overlap, тим більше чанків, а значить більше ембеддингів і більше витрат.

## Частина 5. Hybrid search

У `06_hybrid_search.py` я реалізував три варіанти пошуку:

1. BM25;
2. vector search через Pinecone;
3. hybrid search через RRF.

BM25 рахується локально по:

```text
title + abstract
```

Vector search йде через Pinecone. Потім два списки об'єднуються через Reciprocal Rank Fusion:

```text
RRF(d) = sum(1 / (k + rank(d)))
```

Я використав:

```python
RRF_K = 60
```

### Запит 1: `BERT fine-tuning`

| Метод | №1 результат | Скор |
|---|---|---|
| BM25 | Alzheimer's disease detection based on large language model prompt… | 15.725 |
| Vector | Reinforcement Learning for Flexibility Design Problems | 0.853 |
| Hybrid | Alzheimer's disease detection based on large language model prompt… | 0.01639 |

Для цього запиту BM25 спрацював зрозуміліше, бо він добре ловить точне слово `BERT`. Векторний пошук дав менш релевантний результат.

### Запит 2: `Yann LeCun convolutional networks`

| Метод | №1 результат | Скор |
|---|---|---|
| BM25 | Depth-Adaptive Computational Policies for Efficient Visual Tracking | 14.273 |
| Vector | Handwritten Indic Character Recognition using Capsule Networks | 0.880 |
| Hybrid | Handwritten Indic Character Recognition using Capsule Networks | 0.03178 |

Тут BM25 краще реагує на точні слова, а vector search краще ловить загальну тему convolutional networks. Hybrid підняв результат, який був високо у векторному пошуку і також був у BM25.

### Запит 3: `making computers understand human emotions from text`

| Метод | №1 результат | Скор |
|---|---|---|
| BM25 | Text2Avatar: Text to 3D Human Avatar Generation… | 18.318 |
| Vector | An affective computational model for machine consciousness | 0.872 |
| Hybrid | An Attentive Sequence Model for Adverse Drug Event Extraction… | 0.03083 |

Для цього запиту векторний пошук виглядає кориснішим, бо запит сформульований не як точний термін, а як опис ідеї. BM25 більше чіпляється за окремі слова типу `text`.

### Загальний висновок по hybrid search

BM25 краще працює, коли в запиті є точні терміни, назви моделей або прізвища. Наприклад, `BERT` або `LeCun`.

Векторний пошук краще працює, коли запит написаний звичайною мовою і не обов'язково містить ті самі слова, що й стаття.

Hybrid search через RRF — це компроміс. Він не завжди дає ідеальний топ-1, але часто піднімає документи, які непогано виглядають в обох підходах.

Якщо зменшити `k` в RRF, перші місця будуть мати набагато більшу вагу. Якщо збільшити `k`, різниця між позиціями буде менша, і результат стане більш згладженим.

## Частина 6. Висновки

### BM25 vs vector search

BM25 добре підходить для точного пошуку по словах. Наприклад, якщо користувач шукає конкретну модель, термін або автора.

Vector search краще підходить для смислового пошуку. Наприклад, коли запит сформульований як опис: `making computers understand human emotions from text`.

Тому для реальної системи я б не обирав тільки один варіант. Найкраще використовувати hybrid search, бо він поєднує сильні сторони двох підходів.

### Розмір чанків

Занадто маленькі чанки погані тим, що в них мало контексту. Модель може не зрозуміти, про що йде мова.

Занадто великі чанки теж не дуже хороші, бо в одному векторі змішується багато різних думок.

Для abstract мені здається нормальним розмір приблизно 50–150 слів. У цій роботі я використав 100 слів для fixed-size і до 150 слів для semantic chunking.

### Euclidean distance на нормалізованих векторах

Euclidean distance не зламає пошук, якщо вектори нормалізовані. У такому випадку L2 і cosine дають однаковий порядок найближчих документів.

Але якщо вектори не нормалізовані, тоді L2 може працювати інакше, бо буде враховувати не тільки напрямок, а й довжину вектора.

### Обмеження Pinecone Starter і масштабування

Для 10 000 векторів Pinecone Starter вистачає. Але якщо даних буде набагато більше, наприклад 10 млн статей, з'являться проблеми:

- треба більше місця для зберігання векторів;
- ембеддинги треба рахувати батчами і бажано на GPU;
- повні тексти краще зберігати окремо, а в Pinecone тримати тільки `id` і короткі metadata;
- можливо, треба буде перейти на платний тариф або інше рішення, наприклад Qdrant;
- варто подумати про квантизацію або інші способи зменшення розміру векторів.

Тобто основна проблема при масштабі 10 млн — не сам код пошуку, а інфраструктура, пам'ять, ціна і час обчислення ембеддингів.
